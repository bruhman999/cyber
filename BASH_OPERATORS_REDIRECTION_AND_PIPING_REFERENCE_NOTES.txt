BASH OPERATORS, REDIRECTION & PIPING — REFERENCE NOTES
=======================================================

CORE OPERATORS
--------------
Operator   Meaning                                          Example
--------   -------                                          -------
&&         Run next ONLY if previous succeeded              mkdir test && cd test
;          Run next regardless of success or failure        echo hi; pwd
|          Pipe output of left into right command           ls | grep txt
||         Run next ONLY if previous FAILED                 cd test || echo "failed"

MKDIR + CD PATTERNS
-------------------
# Wrong way — pipe doesn't work here
mkdir testing | cd testing        # BROKEN — cd gets no input, runs in subshell

# Correct ways:
mkdir testing && cd testing       # Only cd if mkdir succeeded
mkdir testing; cd testing         # Always cd even if mkdir failed
mkdir -p testing && cd testing    # -p = no error if exists, create parent dirs too

# Nested directory example:
mkdir -p projects/python/flask && cd projects/python/flask

REDIRECTION OPERATORS
---------------------
Operator   Meaning                                          Example
--------   -------                                          -------
>          Redirect stdout to file (overwrite)              echo hi > file.txt
>>         Redirect stdout to file (append)                 echo hi >> file.txt
2>         Redirect stderr (errors) to file                 cmd 2> errors.txt
2>&1       Redirect stderr INTO stdout stream               cmd > out.txt 2>&1
&>         Redirect both stdout and stderr                  cmd &> all_output.txt
/dev/null  Black hole — discard output entirely             cmd > /dev/null

PROCESS SUBSTITUTION — >(command) and <(command)
-------------------------------------------------
>( )  = treat a command as if it were a file to write TO
<( )  = treat a command as if it were a file to read FROM

This lets you send output to MULTIPLE places at once.

EXAMPLE 1 — Send errors to less AND save them:
-----------------------------------------------
example > >(less) 2> >(tee errors.txt)

Breakdown:
  example              runs the command
  >  >(less)           stdout goes into less (paginated viewer)
  2> >(tee errors.txt) stderr goes into tee which:
                         - prints errors to terminal
                         - saves errors to errors.txt simultaneously

Flow:
  example
    ├── stdout ──────────────► less (view output page by page)
    └── stderr ──────────────► tee
                                ├── prints to terminal
                                └── saves to errors.txt

EXAMPLE 2 — Copy directory to MULTIPLE remote machines simultaneously:
----------------------------------------------------------------------
tar cf - directory | tee >(ssh neo tar xf -) >(ssh 10.0.0.124 tar xf -) > /dev/null

Breakdown:
  tar cf - directory          creates tar archive of directory, outputs to stdout (-)
  |                           pipes that tar stream into tee
  tee                         splits the stream into multiple outputs simultaneously
  >(ssh neo tar xf -)         sends stream to first machine (neo), extracts there
  >(ssh 10.0.0.124 tar xf -)  sends stream to second machine (10.0.0.124), extracts there
  > /dev/null                 discards tee's own stdout (we don't need it locally)

Flow:
  tar archive
       │
       ▼
      tee
       ├──► ssh neo          → extracts on machine 1
       ├──► ssh 10.0.0.124   → extracts on machine 2
       └──► /dev/null        → discarded

Real world use: clone a directory to two servers in one command with no temp file needed.

TEE COMMAND
-----------
tee reads stdin and writes to BOTH stdout AND a file simultaneously.

  command | tee output.txt          # print to terminal AND save to file
  command | tee -a output.txt       # same but append instead of overwrite
  command | tee file1.txt file2.txt # write to multiple files at once

PIPING EXAMPLES
---------------
# Filter output
ls | grep txt                        # list only .txt files
ps aux | grep firefox                # find firefox process
cat /var/log/auth.log | grep Failed  # find failed logins

# Count lines
ls | wc -l                           # count files in directory
cat file.txt | wc -l                 # count lines in file

# Chain multiple pipes
ps aux | grep ssh | awk '{print $2}' # get PID of ssh process

# Send find output to grep
find . -name "*.txt" | grep notes

FIND COMMAND
------------
find . -name "*.txt"                 # find all .txt files in current dir
find /var/log -name "*.log"          # find all logs in /var/log
find . -name "*.sh" | xargs chmod +x # make all shell scripts executable

COMMAND SUBSTITUTION — $(...)
-----------------------------
$(...) runs a command and substitutes its OUTPUT into another
command, as if you'd typed the result directly.

  echo $(date)                          # prints today's date
  sudo docker stop $(sudo docker ps -aq)  # runs "docker ps -aq" first,
                                           # then feeds those container
                                           # IDs in as arguments to "stop"

Just like &&, ;, |, and ~ (home dir expansion), $(...) is SHELL
SYNTAX — a program has to actually be running through a shell to
understand it. This matters a lot for autostart configs (see below).


EXEC-ONCE / AUTOSTART CONFIGS AND SHELL SYNTAX (e.g. Hyprland)
----------------------------------------------------------------
Many configs that "run a command for you" — like Hyprland's
"exec-once =", systemd's ExecStart=, or a .desktop file's Exec=
line — do NOT automatically start a shell. They execute a
program directly, the same way double-clicking an icon would.

  exec-once = obs        -> just runs the "obs" program. Fine —
                            no shell syntax involved.

  exec-once = cd ~/searxng && podman compose up -d
                         -> BROKEN in many setups. This line
                            contains:
                              cd     -> a shell BUILTIN, not a
                                        real executable
                              &&     -> shell syntax
                              ~      -> shell expansion (home dir)
                            None of these exist without a shell
                            interpreting them first.

  FIX — explicitly invoke a shell to interpret the string:
    exec-once = bash -c 'cd ~/searxng && podman compose up -d'
    exec-once = sh -c 'cd ~/searxng && podman compose up -d'

    bash / sh  -> the shell program itself
    -c         -> "treat the next string as a command to run"
    '...'      -> the actual shell command, quoted so the config
                  file passes it through as ONE argument

  RULE OF THUMB: if your command contains &&, ;, |, ||, $(...),
  ~, wildcards (*), or variables ($HOME, $USER, etc.), wrap it in
  bash -c '...' (or sh -c '...') when using it in a config that
  doesn't guarantee a shell — otherwise those symbols won't mean
  anything and the command will likely fail or behave oddly.

  NOTE: some tools DO pass exec lines through /bin/sh internally
  even without you asking (this is implementation-specific and
  can vary by version) — so a shell-syntax line might "just work"
  sometimes. Using bash -c explicitly is more predictable and
  portable regardless of that internal behavior.

  BETTER OPTION FOR SERVICES: if something needs to run reliably
  in the background (like a server/container stack), a systemd
  user service is generally more robust than tying it to
  exec-once in a graphical session config — it can start at
  login (or even boot, with lingering enabled) independent of
  whether your compositor/DE is running at all.


QUICK RULE OF THUMB
-------------------
&&  = "then, if successful"
;   = "then, regardless"
|   = "feed output into next command"
>   = "save output to file"
2>  = "save errors to file"
>() = "send output into a command as if it were a file"
tee = "split output — send to terminal AND somewhere else simultaneously"
$() = "run a command and drop its output right here"
bash -c '...' / sh -c '...' = "force shell syntax to work in
                                configs that don't start a shell"
